# Contributing to `hlib`

Thanks for taking the time to contribute! `hlib` is a Helm **library chart**, so most
changes touch reusable templates, their documentation, and the values schema. This guide
explains how to set up your environment, run checks locally, and open a pull request that
passes CI on the first try.

By participating in this project you agree to keep interactions respectful and
constructive. Be kind, assume good intent, and keep discussions focused on the work.

## Ways to contribute

- Report bugs and unexpected template output.
- Improve or fix existing templates and helpers in `charts/hlib/templates/`.
- Add or update documentation in `charts/hlib/.docs/` and `values.yaml`.
- Add new examples under `charts/hlib/examples/`.
- Propose new resource templates or helper functions.

If you are planning a larger change, please open an issue first so the approach can be
discussed before you invest significant effort.

## Development setup

You will need the following tools installed locally. The versions below are the minimum
recommended; newer releases are generally fine.

| Tool | Purpose | Install |
| --- | --- | --- |
| [Helm](https://helm.sh/docs/intro/install/) `>= 3.x` | Render, lint, and package charts | `brew install helm` |
| [helm-docs](https://github.com/norwoodj/helm-docs) | Generate `README.md` files from templates and `values.yaml` | `brew install norwoodj/tap/helm-docs` or `go install github.com/norwoodj/helm-docs/cmd/helm-docs@latest` |
| [helm-values-schema-json](https://github.com/losisin/helm-values-schema-json) | Generate `values.schema.json` | `helm plugin install https://github.com/losisin/helm-values-schema-json` |

Clone the repository and verify your toolchain:

```shell
git clone git@github.com:anatolek/helm-charts.git
cd helm-charts

helm version
helm-docs --version
helm schema --help
```

## Repository layout

```text
charts/hlib/
  templates/        # Library templates (named templates, prefixed _underscore files)
  .docs/            # Per-feature documentation fragments, assembled into README.md
  examples/         # Consumer charts demonstrating real usage
  values.yaml       # Default values + inline documentation used by helm-docs
  values.schema.json# Generated JSON schema (do not edit by hand)
  README.md         # Generated by helm-docs (do not edit by hand)
  README.md.gotmpl  # Template that assembles the chart README
docs/               # Project-level guides (e.g. BEST-PRACTICES.md)
README.md           # Repository landing page
```

Because `hlib` is a `type: library` chart, its templates are not rendered on their own.
They are exercised through the consumer charts in `charts/hlib/examples/`.

## Running checks locally

CI runs the same checks that you can run locally before pushing. Running them yourself
avoids round-trips with the `fail-on-diff` jobs.

### 1. Render the examples

Each example is a real chart that depends on the library. Render them to confirm your
template changes produce valid manifests:

```shell
cd charts/hlib/examples/simple-deployment
helm template test .
```

Repeat for any example affected by your change (`cronjob`, `rbac`, `volumes`,
`multiple-deployments`, `init-deployment`, `env-variables`, `subcharts`, ...). When you
add behavior, prefer adding or extending an example so it stays covered.

### 2. Lint

```shell
cd charts/hlib/examples/simple-deployment
helm lint .
```

### 3. Regenerate documentation

Run from the repository root so all `README.md` files are refreshed:

```shell
helm-docs --skip-version-footer
```

### 4. Regenerate the values schema

```shell
cd charts/hlib
helm schema --use-helm-docs
```

After running steps 3 and 4, `git status` should be clean except for the changes you
intended. The CI jobs **Generate Helm documentation** and **Generate values schema json**
fail when the committed output differs from a fresh generation.

## Code conventions

- Template files in `charts/hlib/templates/` are partials and are prefixed with an
  underscore (e.g. `_deployment.yaml`). They define named templates such as
  `hlib.deployment`, never standalone manifests.
- Namespace all named templates under the `hlib.` prefix to avoid collisions in consumer
  charts.
- Keep templates configurable through `values.yaml` and the documented `context` /
  `values` / `override` parameters rather than hardcoding values.
- Provide sensible defaults so a minimal consumer configuration renders a working
  manifest.
- Use 2-space indentation in YAML and keep generated manifests deterministic (avoid
  random ordering, stable map iteration with `sortAlpha` where relevant).
- Match the style of the surrounding code and existing templates.

## Documentation changes

`README.md` files (both repository-level chart docs and example docs) are **generated** by
helm-docs. Do not edit them directly — your changes would be overwritten and CI would
fail.

- Document values by adding helm-docs comments in `values.yaml`.
- Add a new documentation section as a separate file under `charts/hlib/.docs/`, then link
  it from `charts/hlib/README.md.gotmpl`.
- Regenerate with `helm-docs --skip-version-footer` and commit the generated output
  together with your source changes.

## Values schema changes

`charts/hlib/values.schema.json` is generated by the
[helm-values-schema-json](https://github.com/losisin/helm-values-schema-json) plugin from
`values.yaml`. Do not edit it by hand.

```shell
cd charts/hlib
helm schema --use-helm-docs
```

Commit the regenerated `values.schema.json` with the `values.yaml` change that produced it.

## Versioning rules

The chart version in `charts/hlib/Chart.yaml` follows semantic versioning `X.Y.Z`:

- `Z` (patch) — increase **for fixes**.
- `Y` (minor) — increase for updates that **add new features** in a backward-compatible
  way.
- `X` (major) — increase when a change **breaks backward compatibility**.

Bump the version in the same pull request that introduces the change, and add a matching
entry to the `artifacthub.io/changes` annotation in `Chart.yaml` describing the change
(`kind: added` / `changed` / `fixed` / `deprecated` / `removed`). Releases are published
automatically from `main` by the **Release Charts** workflow.

## Commit and branch conventions

- Branch from `main`. Branch names follow the `<issue-number>-<short-description>` pattern
  (e.g. `16-add-contributingmd`).
- Write clear, imperative commit messages that describe the change and the "why".
- Group related changes together; keep unrelated changes in separate commits/PRs.

## Pull request expectations

Before opening a PR, make sure that:

- [ ] Affected examples render successfully (`helm template`) and lint cleanly
      (`helm lint`).
- [ ] Documentation is regenerated (`helm-docs --skip-version-footer`) and committed.
- [ ] `values.schema.json` is regenerated (`helm schema --use-helm-docs`) and committed.
- [ ] The chart version and `artifacthub.io/changes` annotation are updated when behavior
      changes.
- [ ] The PR description explains the motivation, the change, and any user-facing impact;
      link the related issue.

CI runs the documentation and schema diff checks on every pull request and **fails on any
diff**. Keep the PR scope focused to make review easier.

## Reporting bugs and requesting features

Open an issue on the
[GitHub issue tracker](https://github.com/anatolek/helm-charts/issues). For bugs, include:

- The library version (`charts/hlib/Chart.yaml`) and your Helm version.
- A minimal `values.yaml` and template include that reproduces the problem.
- The rendered output you got versus what you expected.
